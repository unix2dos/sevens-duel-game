# 排七 Web 展示包装设计文档：音效与 README 升级

## 1. 背景

当前项目已经具备完整的单机对局流程、AI 难度分层和 PixiJS 牌桌表现，但对外展示仍有两个明显缺口：

- 游戏已经接入 `howler` 与音效触发点，却没有任何真实音频资源，试玩时是无声的。
- [README.md](/Users/liuwei/.codex/worktrees/d051/sevens-duel-game/README.md) 仍停留在开发说明页，没有游戏内截图，也没有能证明真实动效和玩法的演示入口。

这两个问题叠加后，项目给仓库访客的第一印象仍然偏“可运行的原型”，而不是“值得点开试玩的完整作品”。

## 2. 产品目标

本轮工作的目标不是扩玩法，而是做一轮展示包装，让仓库访客在 10 秒内完成三件事：

- 看懂这是什么游戏。
- 感受到它已经有产品质感，而不是脚手架 demo。
- 愿意点开试玩或查看演示视频。

## 3. Definition of Done

当以下条件同时成立时，本轮升级视为完成：

- 游戏内存在真实可播放的音效资源，至少覆盖按钮交互、开局/再来一局、出牌、补牌、胜负收尾这些关键事件。
- 音效整体风格偏电影化展示感，但保持克制，不加入循环背景音乐。
- README 前半屏能清楚说明项目定位、核心亮点，并包含真实游戏截图。
- README 提供一个 GitHub 生态内可访问的演示视频入口，用于展示真实玩法和动效。
- 允许为了截图和演示效果做小幅产品化润色，但不修改核心玩法、规则和 AI 逻辑。
- 素材、路径、许可说明、构建和测试状态可以一起交付，不留下无来源资源或失效链接。

## 4. 产品结论

基于需求访谈，这次升级的产品结论如下：

- 主要服务对象是 `仓库访客`，而不是深度玩家。
- README 是主舞台，游戏内音效负责支撑试玩和演示视频的成品感。
- README 以 `中文为主`。
- 演示视频入口必须落在 `GitHub 生态内`。
- 展示素材采用 `一次性产出并入仓` 的方式，不要求命令化复现。
- 音效风格选择 `电影化展示感`，但只做 `SFX + 短提示音`，不做循环 BGM。
- 允许做 `小幅产品化润色`，不做大规模界面重构。
- 音效资源可使用 `可商用 / 可再分发` 的现成素材，并在仓库内处理许可信息。

## 6. 游戏内音效方案

### 6.1 设计目标

音效的职责不是“页面每次变化都响一下”，而是提升关键动作的感知强度，让试玩和演示视频更像成品。

听感目标：

- 比纯拟真更抓人。
- 比街机反馈更克制。
- 保留牌桌材质感和短促仪式感。

### 6.2 声音分层

建议将声音分为三层：

- `界面层`
  - 覆盖首页按钮、规则弹窗、音效开关。
  - 目标是确认操作已发生，声音最轻、最短。
- `牌局动作层`
  - 覆盖开局进入、玩家出牌、AI 出牌、补牌。
  - 目标是建立材质感与动作节奏，是最常被听到的一层。
- `收尾层`
  - 覆盖结果页进入和再来一局触发。
  - 允许略带戏剧性，但时长必须短，不拖慢流程。

### 6.3 关键触发点

本轮建议明确保留以下触发：

- 首页点击“开始游戏”：播放较有仪式感的启动提示音。
- 首页普通按钮和开关：播放轻确认音。
- 新开一局或再来一局触发：播放开局 / 发牌音。
- 玩家成功出牌：播放主确认音。
- AI 出牌：沿用出牌音，但强度略低于玩家动作。
- 玩家补牌：播放区别于出牌的补牌音。
- 自动补牌或儿童模式自动动作：仍发声，但强度比玩家主动动作更收敛。
- 对局结束进入结算：播放短促收尾音。

### 6.4 明确不发声的地方

以下变化不加音效，避免廉价感和疲劳：

- 普通文本刷新
- 事件流更新
- 鼠标悬停
- 非关键动画中间帧
- 一般状态切换提示

### 6.5 节奏原则

- 同一时刻只保留一个主音效占主导。
- 玩家主动动作优先级高于系统自动动作。
- 开局和结算可以轻微“抬情绪”，中盘动作必须收着。
- 高频动作短促，低频关键节点可略长。
- 音量层级保持 `收尾层 > 牌局动作层 > 界面层`，但差距不能夸张。

## 7. README 展示方案

### 7.1 首页目标

README 首屏需要先回答：

- 这是什么游戏？
- 它的完成度和气质是什么？
- 为什么值得继续看或点开试玩？

因此 README 必须从“开发说明页”升级为“作品展示页 + 开发入口”的双层结构。

### 7.2 信息架构

推荐顺序如下：

1. 项目名 + 一句话定位
2. 核心卖点短列表
3. 游戏截图区
4. 演示视频入口
5. 玩法 / 特性简述
6. 开发、测试、构建命令

这样可以保证仓库访客先看到价值和证据，再进入工程信息。

### 7.3 截图区策略

截图至少包含三张真实界面：

- `首页`
  - 负责传达定位和气质。
- `对局中`
  - 负责证明玩法真实存在，且牌桌表现足够好。
- `结果页`
  - 负责展示完整回路，而非只有首页壳。

截图原则：

- 使用真实运行画面，不用概念拼图。
- 构图应适合 README 对外展示。
- 图片本身尽量少叠说明文字，解释留给 README 文案。

### 7.4 视频入口策略

视频入口采用 `GitHub 内文件页面` 的方式：

- 将短演示视频作为仓库可访问资源提交。
- README 通过封面图或配图链接跳转到视频文件页面。
- 视频内容必须展示真实玩法和动效，而不是静态轮播或口播介绍。

## 8. 技术实现策略

### 8.1 音频实现

技术上不重建音频系统，直接复用现有结构：

- [src/audio/sounds.ts](/Users/liuwei/.codex/worktrees/d051/sevens-duel-game/src/audio/sounds.ts)
  - 补全资源路径、音量基线和必要的多资源候选。
- [src/audio/useSound.ts](/Users/liuwei/.codex/worktrees/d051/sevens-duel-game/src/audio/useSound.ts)
  - 补足初始化稳定性、空资源兜底和浏览器音频解锁。
- [src/App.tsx](/Users/liuwei/.codex/worktrees/d051/sevens-duel-game/src/App.tsx)
  - 继续作为主要音效编排中心，集中定义触发语义。

运行时音频文件建议放在 `public/assets/audio/`，由 Vite 直接静态分发。

### 8.2 README 素材实现

README 专用素材与游戏运行资源分开管理：

- 游戏运行时资源：`public/assets/audio/`
- README 素材：`docs/assets/showcase/`

这样一眼能区分“运行依赖”和“文档展示资产”，后续替换也更清楚。

### 8.3 许可说明

增加单独素材说明文件，记录：

- 音效名称
- 来源链接或来源说明
- 许可类型
- 是否需要署名

避免把来源信息零散写进 README 主正文。

## 9. 页面润色边界

本轮允许做少量面向展示的润色，但改动范围必须收敛：

- 首页
  - 强化一句话定位和行动区层次。
- 对局页
  - 保持现有牌桌结构，只保证截图时状态和信息更统一。
- 结果页
  - 优化文案与 CTA，让结果页更像完整结束态。

不做整页布局重写，不引入演示专用模式。

## 10. 验证方案

### 10.1 功能验证

- `npm run build`
- `npm run test`
- `npm run test:e2e`

保证音频与 README 改动不破坏既有构建和基础烟测。

### 10.2 体验验证

手动检查以下事实：

- 首次用户交互后音频可正常播放。
- 音效开关能全局生效。
- 关键事件触发与声音类型一一对应。
- 连续出牌和补牌不会爆音、失真或明显错位。

### 10.3 展示验证

- README 图片路径全部有效。
- README 视频入口可点击并跳到 GitHub 内视频页面。
- README 前半屏在不看开发命令的情况下仍能理解项目价值。

## 11. 风险与缓解

### 11.1 音频授权不清

缓解方案：将来源与许可单独落地，不接受来源不明文件入仓。

### 11.2 浏览器音频策略导致首次无声

缓解方案：在首个用户交互后完成音频初始化或解锁，避免自动播放策略拦截。

### 11.3 README 展示素材与真实产品脱节

缓解方案：所有截图和视频都基于真实运行页面产出，不做概念图拼装。

## 12. 结论

本轮升级的本质不是“加几个资源文件”，而是把项目从工程演示状态推进到可对外展示的作品状态。README 负责第一眼的理解和转化，音效负责试玩和演示视频的质感支撑，两者共同服务同一个目标：让仓库访客愿意继续看、继续点、继续玩。
